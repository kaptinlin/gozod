# 会议内容提取

从 `.meetings/*.md` 会议记录文件中提取与当前项目相关的内容，过滤掉无关讨论，输出清理后的项目相关信息。

## 使用场景

- 会议记录包含多个项目或话题，需要提取特定项目的相关内容
- 会议记录冗长，需要快速定位与当前任务相关的讨论
- 开始新任务前，需要从历史会议中提取背景信息
- 需要为特定功能或模块收集相关的决策和讨论

## 输入要求

**会议记录文件位置**：`.meetings/*.md`

会议记录应包含：
- 讨论的主题和内容
- 技术决策和方案
- 行动项和待办事项
- 相关的背景信息

## 提取流程

### 1. 明确项目范围

在开始提取前，确认：
- 当前项目的名称和范围
- 需要关注的模块、功能或技术栈
- 相关的关键词和术语

如果用户未明确说明，询问：
- "你关注的项目或功能是什么？"
- "需要提取哪些方面的内容？（如架构、实现细节、待办事项等）"

### 2. 读取会议记录

```bash
ls .meetings/*.md
```

按时间顺序或相关性读取会议记录文件。优先读取：
- 文件名包含项目关键词的记录
- 最近的会议记录（通常文件名以日期开头）

### 3. 识别相关内容

扫描会议记录，识别与项目相关的段落。相关内容通常包括：

**直接相关**：
- 明确提到项目名称、模块名称或功能名称
- 讨论项目的技术方案、架构设计
- 项目相关的决策和结论
- 项目的行动项和待办事项

**间接相关**：
- 影响项目的基础设施或依赖变更
- 相关技术栈的选型和最佳实践
- 团队规范和流程调整

**无关内容**（应过滤）：
- 其他项目的讨论
- 与项目无关的行政事务
- 无关的技术话题

### 4. 提取和整理

将相关内容按以下结构整理：

```markdown
# [项目名称] 相关内容提取

## 来源
- 会议记录：[文件名]
- 提取时间：[日期]

## 核心决策

[提取的关键决策，保留原文或改写为简洁表述]

## 技术方案

[提取的技术方案讨论，包括方案对比、选型原因]

## 实现细节

[提取的实现相关讨论，如架构设计、模块划分、接口定义等]

## 待办事项

[提取的行动项和待办任务]

## 背景信息

[提取的相关背景，如问题描述、需求来源、约束条件等]
```

### 5. 输出格式

**简洁模式**（默认）：
- 只保留核心信息
- 使用简洁的语言改写
- 去除冗余和重复内容
- 适合快速了解项目状态

**详细模式**（用户明确要求时）：
- 保留更多上下文
- 包含讨论过程和权衡
- 保留原文的技术细节
- 适合深入理解决策背景

## 提取原则

### 保持准确性

- 不改变原意，不添加推测
- 保留关键术语和技术名词
- 如有疑问，保留原文或标注不确定

### 聚焦相关性

- 严格过滤无关内容
- 优先提取决策和结论
- 次要提取讨论过程
- 最后提取背景信息

### 结构清晰

- 使用清晰的标题组织内容
- 相同主题的内容归类在一起
- 保持逻辑顺序（背景→讨论→决策→行动）

### 可追溯性

- 标注内容来源（会议记录文件名）
- 保留关键人物的观点（如有必要）
- 记录决策的时间点

## 常见模式

### 多文件提取

当需要从多个会议记录中提取内容时：

```markdown
# [项目名称] 相关内容汇总

## 来源
- 20240301-架构讨论.md
- 20240308-技术方案评审.md
- 20240315-进展同步.md

## 时间线

### 2024-03-01 架构讨论
[提取的内容]

### 2024-03-08 技术方案评审
[提取的内容]

### 2024-03-15 进展同步
[提取的内容]

## 综合结论
[跨会议的总结和关联]
```

### 按主题提取

当会议记录包含多个主题时：

```markdown
# [项目名称] 相关内容提取

## 架构设计
[从不同会议中提取的架构相关内容]

## 技术选型
[从不同会议中提取的选型相关内容]

## 实现计划
[从不同会议中提取的实现相关内容]
```

### 问题导向提取

当需要回答特定问题时：

```markdown
# 关于 [具体问题] 的会议内容提取

## 问题
[用户关心的问题]

## 相关讨论
[从会议记录中提取的相关讨论]

## 结论
[基于会议内容的结论]

## 待解决
[仍未解决的问题]
```

## 输出位置

提取的内容可以：
- 直接在对话中展示（默认）
- 保存为新文件（用户要求时）：`.meetings/extracted/[项目名]-[日期].md`
- 追加到现有文档（用户指定时）

## 质量检查

提取完成后，检查：

- ✅ 是否遗漏了重要的决策或讨论
- ✅ 是否包含了无关内容
- ✅ 内容是否准确反映原文
- ✅ 结构是否清晰易读
- ✅ 是否标注了来源

## 示例

**用户请求**：
"从最近的会议记录中提取关于用户认证模块的讨论"

**执行步骤**：

1. 列出会议记录文件
2. 读取最近的 3-5 个会议记录
3. 搜索包含"用户认证"、"登录"、"鉴权"等关键词的段落
4. 提取相关段落并按主题整理
5. 输出结构化的提取结果

**输出示例**：

```markdown
# 用户认证模块相关内容提取

## 来源
- 20240301-后端架构讨论.md
- 20240308-安全评审.md

## 核心决策

采用 JWT + Refresh Token 方案，Access Token 有效期 15 分钟，Refresh Token 有效期 7 天。

## 技术方案

### 认证流程
1. 用户登录，验证用户名密码
2. 生成 Access Token 和 Refresh Token
3. Access Token 存储在内存，Refresh Token 存储在 HttpOnly Cookie
4. Token 过期后使用 Refresh Token 刷新

### 安全考虑
- 密码使用 bcrypt 加密，cost factor 设为 12
- 实施 rate limiting，防止暴力破解
- 添加设备指纹验证，检测异常登录

## 待办事项
- [ ] 实现 Token 刷新接口
- [ ] 添加登录日志记录
- [ ] 实现多设备管理功能
```
