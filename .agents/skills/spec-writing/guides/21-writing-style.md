# 写作风格

## Overview

定义 SPEC 的写作风格和语言规范。不涵盖内容边界（见 [20-content-boundaries.md]）、代码示例（见 [24-code-examples.md]）、AI 可读性优化（见 [25-ai-readability.md]）。

目标：简洁、精确、可操作。每句话都有价值，无冗余文字。

## 简洁原则

### 每句话都有价值

删除不影响理解或决策的内容。

**好的做法**:
```markdown
## 错误包装
使用 `fmt.Errorf("%w: context", err)` 包装错误。
始终使用 %w 以支持 errors.Is() 链。
```

**不好的做法**:
```markdown
## 错误包装
错误处理是 Go 的关键部分。当错误发生时，提供上下文对调试很重要。
这有助于追踪问题来源。在我们的代码库中，我们遵循标准的 Go 错误
包装模式，使用 fmt.Errorf 和 %w 动词。%w 动词很重要，因为它允许
errors.Is() 正确工作...
```

> **Why**: 冗长的文字掩盖了关键信息。工程师需要快速找到规则，而非阅读动机性文字。

**测试方法**: 删除一句话，规则是否仍然清晰？如果是，删除它。

### 先规则，后依据

规则在前，决策依据在后（`> **Why**` 块）。

**好的做法**:
```markdown
## 严重性级别
三个级别：ERROR, WARN, INFO。

> **Why**: 与 Go `log/slog` 和 POSIX syslog 对齐。三个级别足够——
> 五个级别会造成歧义（这是 WARN 还是 CAUTION？）。DEBUG 是运行时关注点，
> 不适用于静态分析。
```

**不好的做法**:
```markdown
## 严重性级别
我们需要一个清晰的严重性系统。经过研究，我们发现大多数日志系统使用
三到五个级别。考虑到我们的用例，我们决定使用三个级别：ERROR, WARN, INFO。
这与 syslog 对齐...
```

> **Why**: 工程师首先需要知道"是什么"，然后才关心"为什么"。叙事性开头浪费时间。

### 删除填充词

避免无意义的修饰词和过渡短语。

| 填充词 | 简洁替代 |
|--------|---------|
| "为了实现这个目标" | 删除，直接说明目标 |
| "需要注意的是" | 删除，直接陈述事实 |
| "基本上"、"实际上" | 删除 |
| "我们决定使用 X" | "使用 X" |
| "应该考虑" | "考虑" 或删除 |

**示例**:
```markdown
<!-- 不好 -->
为了确保一致性，我们需要注意的是，基本上所有的 API 实际上都应该考虑使用 JSON。

<!-- 好 -->
所有 API 使用 JSON。
```

## 精确动词选择

### 使用具体动词

避免模糊动词，使用精确的领域动词。

| 模糊动词 | 精确替代 | 示例 |
|---------|---------|------|
| 处理（handle） | 验证、转换、存储、发送 | "验证输入" 而非 "处理输入" |
| 管理（manage） | 创建、更新、删除、查询 | "创建用户" 而非 "管理用户" |
| 执行（do/perform） | 计算、渲染、发送、保存 | "计算折扣" 而非 "执行计算" |
| 获取（get） | 查询、加载、读取、提取 | "查询订单" 而非 "获取订单" |

> **Why**: 精确动词减少歧义，明确操作的性质。"处理"可能意味着任何事情；"验证"有明确含义。

### 避免模糊名词

禁止使用无意义的技术后缀。

**禁止的后缀**:
- `Manager` — 管理什么？如何管理？
- `Handler` — 处理什么？如何处理？
- `Helper` — 帮助谁？做什么？
- `Util` / `Utils` — 杂物抽屉，职责不清

**替代方案**:
```markdown
<!-- 不好 -->
UserManager — 管理用户（如何管理？）
DataHandler — 处理数据（什么数据？如何处理？）
StringHelper — 帮助字符串（做什么？）

<!-- 好 -->
UserRepository — 存储和查询用户
RequestValidator — 验证请求
StringFormatter — 格式化字符串
```

> **Why**: 模糊名词隐藏了真实职责。如果无法用具体名词命名，说明职责不清晰。

> **Rejected**:
> - 保留 `Manager`多个组件"的标记 — 协调是职责，应该有更具体的名称（如 `Orchestrator`, `Coordinator`）
> - 使用 `Service` 作为通用后缀 — 在服务层中，`Service` 后缀是冗余的（包名已表明）

## 使用祈使句

使用祈使句（命令式），而非被动语态或建议语气。

| 语气 | 示例 | 评价 |
|------|------|------|
| 祈使句（好） | "使用 JSON" | 直接、明确 |
| 被动语态（差） | "JSON 应该被使用" | 冗长、模糊 |
| 建议语气（差） | "建议使用 JSON" | 不够坚定 |

**模糊表达 → 明确替代**：

| 模糊表达 | 替代 |
|---------|------|
| "应该使用 X" | "使用 X" |
| "可能需要 X" | "X" 或删除 |
| "建议 X"、"最好 X"、"尽量 X"、"考虑 X" | "X" 或删除 |

> **Why**: 祈使句消除歧义。"应该"和"建议"暗示可选性，但 SPEC 定义的是必须遵守的规则。

**例外**：`> **Why**` 块使用陈述句（说明事实），而非祈使句。

## 术语一致性

### One Name, One Concept

每个概念只使用一个术语。禁止同义词。

**好的做法**:
```markdown
## 术语表

- **Scope** — 作用域，变量的可见范围
- **Tier** — 层级，token 的抽象层次
- **Layer** — 架构层，系统的水平分层

在整个 SPEC 中：
- 谈论变量可见性时，使用 "scope"
- 谈论 token 抽象时，使用 "tier"
- 谈论系统架构时，使用 "layer"
```

**不好的做法**:
```markdown
<!-- 在不同地方混用术语 -->
Token 有三个 scope... （应该是 tier）
架构分为三个 tier... （应该是 layer）
函数的 layer 是文件级... （应该是 scope）
```

> **Why**: 术语混用造成混淆。读者需要猜测"这里的 scope 是指作用域还是层级？"

> **Rejected**: 使用 "level" 作为通用术语 — 过于模糊，无法区分不同概念

### 术语登记

在命名规范 SPEC 中维护术语表，其他 SPEC 链接到它。

**引用格式**：`见 [文件名 §章节名]`

**引用原则**：
- 定义一次，引用多次
- 引用而非复制

> **Why**: 单一来源避免定义不一致，减少维护成本。

## 避免被动语态

被动语态冗长且模糊主体。

| 被动语态 | 主动语态 |
|---------|---------|
| "JSON 应该被使用" | "使用 JSON" |
| "错误必须被包装" | "包装错误" |
| "验证将被执行" | "验证输入" |

**例外**: 当主体不重要或未知时，被动语态可接受（如"配置文件在启动时加载"）。

## 格式化规范

- 无序列表（`-`）表示并列项；有序列表（`1.`）表示步骤或优先级
- 表格用于结构化对比或映射（决策对比、术语映射、规则矩阵）
- 代码块指定语言（```json, ```go, ```bash），使用注释标注关键点（`✅` / `❌`）

## 抽象层次判断

SPEC 写到什么程度？三个判断标准：

1. **可重新实现测试** — 删除所有代码，仅凭 SPEC 能否重新实现？不能 → 过于抽象；包含实现代码 → 过于具体
2. **稳定性测试** — 6 个月内会改变？会 → 实现细节，不写入 SPEC；不会 → 架构决策，写入 SPEC
3. **决策性测试** — 是"决策"还是"执行"？决策（为什么选这个方案）→ 写入；执行（怎么写代码）→ 不写入

> **Why**: SPEC 是决策文档，不是实现手册。AI 需要知道"选择什么"和"为什么"，而非"如何实现"。

## Terminology

| Term | Definition |
|------|-----------|
| **祈使句** | 命令式语气，直接陈述规则（如"使用 JSON"） |
| **填充词** | 无意义的修饰词和过渡短语（如"基本上"、"实际上"） |
| **模糊动词** | 不精确的动词（如"处理"、"管理"、"执行"） |
| **模糊名词** | 无意义的技术后缀（如 `Manager`、`Handler`、`Helper`） |

## Forbidden

- **Don't write verbose motivational text**: 规则掩盖在叙述中 → 规则在前，依据放 `> **Why**`
- **Don't use "should", "might", "consider"**: 暗示可选性 → 使用祈使句或删除
- **Don't use passive voice (unless necessary)**: 冗长且模糊主体 → 使用主动语态
- **Don't use `Manager`/`Handler`/`Helper` suffixes**: 隐藏真实职责 → 使用 `Repository`, `Validator`, `Formatter`
- **Don't mix synonyms**: 造成混淆 → 在术语表中定义标准术语，全文一致使用
- **Don't write implementation details**: SPEC 定义决策和约束 → 写"什么"和"为什么"，不写"如何"
- **Don't write obvious content**: 浪费阅读时间 → 只保留团队特定规则
- **Don't record change history**: Git 是历史追踪器 → 使用 `> **Why**` 说明当前选择依据
