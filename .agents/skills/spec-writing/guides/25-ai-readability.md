# AI 可读性优化

## Overview

定义如何优化 SPEC 的 AI 可读性。SPEC 的读者包括人类和 AI Agent，写作风格同时优化两者。不涵盖写作风格（见 [21-writing-style.md]）、SPEC 结构（见 [11-structure.md]）。

## 结构化优先

AI 通过识别固定结构和字段来提取信息。使用一致的章节结构和字段名称。

**标准结构模式**：
```markdown
## 功能名称

**用途**：一句话说明
**约束**：
- 约束 1
- 约束 2
**示例**：
```code
示例代码
```
**禁止**：
- 禁止项 1
> **Why**: 决策依据
```

**固定字段**：`**用途**`、`**约束**`、`**示例**`、`**禁止**`、`> **Why**`、`> **Rejected**`

> **Why**: 固定结构让 AI 快速定位关键信息，减少上下文理解成本。

## 关键信息前置

将最重要的约束和规则放在章节开头。先说"必须做什么"，再说"为什么"。

```markdown
<!-- 好 -->
## API 响应格式
使用 JSON。
包含 `Content-Type: application/json` 头。
> **Why**: JSON 是行业标准，工具链完善。

<!-- 差 -->
## API 响应格式
考虑到行业标准和工具链的完善性，我们决定使用 JSON...
```

> **Why**: AI 的注意力机制优先处理章节开头内容。叙事性开头浪费 token。

## 使用清单而非段落

AI 更容易处理列表形式的信息。将约束组织成列表，而非嵌入段落。

```markdown
<!-- 好 -->
必须满足：
- 非空
- 长度 1-100 字符
- 仅包含字母数字和连字符
- 不以连字符开头或结尾

<!-- 差 -->
输入必须非空，长度在 1 到 100 个字符之间，只能包含字母数字字符和连字符，
并且不能以连字符开头或结尾。
```

> **Why**: 列表让每个约束独立可验证。段落形式容易让 AI 遗漏某个约束。

## 明确的引用路径

引用其他 SPEC 时，使用明确的章节路径。

**格式**：
```
[文件名 §章节名]
[文件名 §一级章节 §二级章节]
```

**示例**:
```markdown
<!-- 好 -->
遵循 [12-naming.md §Token命名] 中的三段式命名规则。

<!-- 差 -->
遵循命名规范中的规则。
```

引用放在句子末尾：先陈述规则，再提供引用。这样即使 AI 跳过引用，也能理解核心规则。

> **Why**: 明确的引用路径让 AI 准确定位相关内容。模糊引用导致 AI 猜测或忽略。

## Terminology

| Term | Definition |
|------|-----------|
| **固定字段** | 在所有 SPEC 中一致使用的字段名称（`**用途**`、`**约束**` 等） |
| **关键信息前置** | 将最重要的约束和规则放在章节开头 |
| **引用路径** | 使用 `[文件名 §章节名]` 格式的明确引用 |

## Forbidden

- **Don't use long paragraphs for constraints**: AI 难以提取关键信息 → 使用列表，每个约束一行
- **Don't bury key rules in paragraph middle**: AI 注意力优先处理开头 → 关键信息前置
- **Don't use vague references**: "见命名规范"无法定位 → 使用 `[12-naming.md §Token命名]`
- **Don't mix field names**: 有时"约束"有时"规则" → 使用固定字段名称，全文一致
- **Don't explain before stating rules**: 叙事性开头浪费 token → 先规则，后 `> **Why**`
