# 验收标准

## Overview

定义如何编写验收标准和 BDD 场景，使用 Given-When-Then 格式描述系统行为。验收标准是可验证的规范，明确定义"什么是正确的"，作为人类定义意图、Agent 验证实现的契约。

不涵盖自动化测试框架的使用、CI/CD 配置、测试工具集成。

相关 SPECS：[11-structure.md]（SPEC 结构规范）。

> **Why**: 在 AI 生成代码的时代，人类的价值在于定义"什么是正确的"，而非审查代码实现。验收标准是人类定义意图、机器验证实现的契约。

## Given-When-Then 格式

### 基本结构

```gherkin
Scenario: 场景标题
  Given [前置条件/初始状态]
  When [触发的动作/事件]
  Then [期望的结果/行为]
```

> **Why**: Given-When-Then 提供清晰的因果结构，分离前置条件、触发动作、期望结果。

### 示例：订单价格计算

```gherkin
Scenario: 计算含税订单总价
  Given 订单的不含税价格为 25 欧元
  When 应用 20% 的增值税
  Then 订单的含税总价应为 30 欧元
```

## 场景编写规则

### 规则 1：使用业务语言

场景必须使用业务领域语言，Agent 可理解，非技术人员也能理解。

**好的示例**：
```gherkin
Scenario: 用户登录
  Given 用户"张三"已注册
  When 用户使用正确凭证登录
  Then 登录成功
  And 用户可以访问个人信息
```

**不好的示例**：
```gherkin
Scenario: 用户登录
  Given 数据库中存在 user_id=123 的记录
  When 发送 POST 请求到 /api/login
  Then 返回 HTTP 200 状态码
```

> **Why**: 业务语言让所有人都能理解场景。技术细节属于实现，不属于验收标准。

### 规则 2：场景独立

每个场景必须独立运行，不依赖其他场景的执行顺序或结果。

**好的示例**：
```gherkin
Scenario: 支付订单
  Given 存在一个待支付的订单
  And 订单金额为 100 元
  And 用户账户余额为 500 元
  When 用户支付该订单
  Then 订单状态变为已支付
  And 用户账户余额为 400 元
```

**不好的示例**：
```gherkin
Scenario: 创建订单
  When 用户创建订单 #12345

Scenario: 支付订单（依赖上一个场景）
  When 用户支付订单 #12345
```

> **Why**: 独立场景可以并行执行，可以单独调试，不会因为其他场景失败而失败。

### 规则 3：可验证

场景必须有明确的输入和输出，可以验证是否满足。

**好的示例**：
```gherkin
Scenario: 计算订单折扣
  Given 订单金额为 1000 元
  And 用户是 VIP 会员
  When 应用会员折扣
  Then 订单金额应为 900 元
  And 折扣金额应为 100 元
```

**不好的示例**：
```gherkin
Scenario: 计算订单折扣
  Given 订单金额较高
  When 应用折扣
  Then 订单金额减少
```

> **Why**: 具体数值让场景可验证。模糊的描述无法验证，不同人有不同理解。

## 场景描述规范

### 使用声明式而非命令式

**命令式（不好）**：
```gherkin
Given 用户打开浏览器
And 用户导航到登录页面
And 用户在用户名字段输入"john@example.com"
And 用户点击登录按钮
```

**声明式（好）**：
```gherkin
Given 用户已登录为"john@example.com"
```

> **Why**: 声明式描述关注业务意图，不关注实现细节。

> **Rejected**: 命令式描述 - 包含过多实现细节，场景脆弱，维护成本高。

### 使用具体数值而非变量

**变量（不好）**：
```gherkin
Given 账户余额为 X 元
When 用户取款 Y 元
Then 余额应为 X-Y 元
```

**具体数值（好）**：
```gherkin
Given 账户余额为 1000 元
When 用户取款 300 元
Then 余额应为 700 元
```

> **Why**: 具体数值让场景更易理解，便于验证。

## 边界条件覆盖

### 三类场景

每个功能必须覆盖三类场景：

1. **正常路径（Happy Path）** - 标准使用场景
2. **边界条件（Boundary Cases）** - 空值、最大值、最小值、临界点
3. **异常情况（Exception Cases）** - 错误输入、冲突、权限不足

> **Why**: 边界条件是 bug 的高发区。明确边界行为，避免实现歧义。

### 验证优先级

按以下顺序编写和验证：

| 优先级 | 类型 | 示例 |
|--------|------|------|
| **P0** | 核心功能（必须通过） | 正常路径、核心业务逻辑、数据完整性 |
| **P1** | 边界条件 | 空值、最大值/最小值、临界点 |
| **P2** | 异常情况 | 错误处理、异常恢复、错误信息清晰度 |

> **Why**: P0 失败意味着功能不可用，必须先修复。P1/P2 在 P0 通过后逐步完善。

### 示例：账户余额验证

**正常路径**：
```gherkin
Scenario: 余额充足时取款成功
  Given 账户余额为 1000 元
  When 用户取款 300 元
  Then 取款成功
  And 账户余额为 700 元
```

**边界条件**：
```gherkin
Scenario: 取款金额等于余额
  Given 账户余额为 1000 元
  When 用户取款 1000 元
  Then 取款成功
  And 账户余额为 0 元
```

**异常情况**：
```gherkin
Scenario: 余额不足时取款失败
  Given 账户余额为 100 元
  When 用户取款 300 元
  Then 取款失败
  And 显示错误"余额不足"
```

> **Why**: 边界条件和异常情况是 bug 高发区，必须显式覆盖。

## Terminology

| Term | Definition |
|------|-----------|
| **验收标准 (Acceptance Criteria)** | 判断功能是否正确实现的可验证条件 |
| **Given-When-Then** | BDD 场景格式：前置条件、触发动作、期望结果 |
| **场景独立性** | 每个场景可独立运行，不依赖其他场景 |
| **声明式描述** | 描述"做什么"而非"怎么做" |

## Forbidden

- **Don't 在场景中包含实现细节**: 技术细节属于实现层 → 使用业务语言描述期望行为
- **Don't 编写依赖其他场景的场景**: 无法并行执行和单独调试 → 在 Given 中明确设置所需状态
- **Don't 使用模糊的验收标准**: "金额较高"、"性能满足要求"无法验证 → 使用具体数值和可验证指标
- **Don't 在验收标准中包含测试框架代码**: 测试框架配置属于代码库 → 只编写 Given-When-Then 场景
