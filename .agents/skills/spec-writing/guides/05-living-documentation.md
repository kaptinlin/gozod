# 活文档理念

## Overview

定义活文档的概念层属性：通过一致性机制与代码保持同步的文档特性。

**不涵盖**：工具配置、CI/CD 自动化、团队协作流程。

**相关 SPEC**：[20-content-boundaries.md]（内容边界）、[00-principles.md]（核心原则）。

## 活文档的核心属性

### 1. 与代码同步

SPEC 必须与代码保持同步，不同步时能够被发现。通过一致性机制（测试、类型检查、契约验证）确保 SPEC 与代码描述相同行为。

✅ **好**：类型定义与 SPEC 描述一致，可通过自动化验证
❌ **坏**：纯文字描述"Token 包含 id、name、category 字段"（无可验证定义）

> **Why**: 一致性机制确保 SPEC 与代码同步。当不一致时，验证失败，强制更新。
> **Rejected**: 人工维护同步 — 依赖纪律，容易遗漏。

### 2. 纯文本存储

使用 Markdown，与代码共存于 git 仓库，支持 diff、merge、blame。

✅ **好**：`project/SPECS/*.md` 与 `project/src/` 在同一 git 仓库
❌ **坏**：外部 Google Docs，版本不一致

> **Why**: 纯文本可以使用 git 管理。二进制格式无法 diff。

### 3. 利用已有知识

不重复代码已表达的知识，只提供代码无法表达的内容。

**代码已表达**：数据结构、函数签名、实现细节、依赖关系
**SPEC 应提供**：意图和原理、约束和规则、业务上下文、决策依据

✅ **好**：SPEC 记录命名规则和 Why，类型定义留在代码中
❌ **坏**：重复类型定义中的所有字段说明

> **Why**: 重复记录增加维护成本，代码是最准确的真相来源。

## 活文档的实现方式

### 固有文档优先

对于易变知识，优先使用固有文档（嵌入代码的文档）：类型定义、代码注释、测试用例、配置文件。

SPEC 只记录规则和依据，具体字段定义留在代码的类型系统中。

> **Why**: 固有文档与代码共存，版本一致性自动保证。代码重构时，固有文档随之更新。
> **Rejected**: 所有文档都外部化 — 易变知识会快速过时。

### 按稳定性选择文档化方式

只记录关键场景和稳定知识。根据知识稳定性选择文档化方式。

**稳定知识** → 外部文档（SPEC）：愿景和目标、核心领域概念、架构约束、命名规范
**易变知识** → 固有文档或对话：业务操作细节、技术实现细节、临时决策

✅ **好**：Token 命名规范（跨项目共享，长期稳定，多人需要）
❌ **坏**：如何运行测试（临时操作，应在 README）

> **Why**: 稳定知识的文档化投资可以长期回收。易变知识的文档化成本高于收益。

## 文档困难是设计问题的信号

如果 SPEC 难以编写，可能说明设计有问题。需要记录的特殊知识的量是设计成熟度的指标。

**信号**：场景描述过长 → 缺少抽象；场景过于复杂 → 设计过度耦合；规则难以表达 → 概念不清晰

✅ **好**：3 句话描述清楚，场景简洁（设计清晰）
❌ **坏**：场景包含 10+ 个前置条件，需要大量解释（设计复杂）

> **Why**: 好的设计容易解释。简化设计比简化文档更有价值。
> **Rejected**: 用复杂文档掩盖设计问题 — 治标不治本。

## Terminology

| Term | Definition | Not |
|------|-----------|-----|
| **活文档** | 通过一致性机制与代码保持同步的文档 | Not "静态文档" |
| **一致性机制** | 自动检查文档与代码是否描述相同行为的机制（测试、类型检查、契约验证） | Not "人工审查" |
| **固有文档** | 使用实现技术直接表达知识的文档（代码注释、类型定义） | Not "外部文档" |
| **外部文档** | 与实现技术无关的文档（Markdown、图表） | Not "固有文档" |
| **稳定知识** | 长期不变或变化缓慢的知识 | Not "易变知识" |
| **易变知识** | 频繁变化的知识 | Not "稳定知识" |

## Forbidden

- **Don't rely on manual sync**: 依赖纪律容易遗漏 → 使用一致性机制（测试、类型检查）
- **Don't store SPEC in binary format**: 无法 diff → 使用纯文本（Markdown）与代码共存
- **Don't repeat code-expressed knowledge**: 维护成本高 → SPEC 提供代码无法表达的内容（意图、原理、约束）
- **Don't document everything**: 文档膨胀 → 只记录关键场景和稳定知识
- **Don't externalize volatile knowledge**: 快速过时 → 使用固有文档或对话
- **Don't mask design problems with complex docs**: 治标不治本 → 重构设计，简化系统
