# 记忆模板重构与PDF评估

本次讨论围绕四个核心库的进展展开：记忆库（Memory）的接口重新设计、模板库（Template）的全面重构、Go 文档库向 v3 架构的迁移，以及 PDF 处理方案的技术评估与路径选择。

## 记忆库：合并与淘汰机制重新设计

### 整体架构调整

记忆库移除了之前的"演化"功能，当前聚焦于三个核心能力：生成、合并和过期淘汰。其中合并和生成沿用了之前的接口设计，只是将它们统一迁移到了 memory 模块内部。过期淘汰功能则基于此前定义的一组权重参数来计算——它接收一批记忆数据，经过权重计算后返回处理结果。

### 合并接口的改进

合并功能做了一项关键改进：移除了之前的 T 泛型参数，返回值结构变得更加清晰。现在每条返回的记忆都携带一个明确的状态标识，状态只有三种：

- 保持原样——这条记忆没有发生变化
- 冲突——两条记忆之间存在矛盾
- 合并——多条记忆被归并为一条

无论是冲突还是合并，返回结果中都会通过一个切片字段记录相关记忆的 ID，因为每条记忆数据都有唯一的 ID 标识。合并状态下，这个字段会列出被合并的所有源记忆 ID；冲突状态下，则记录互相冲突的记忆 ID。最终的 memoryData 会以其中一条记忆为主体来呈现。这样一来，调用方可以通过合并接口返回的状态，清楚地知道每条记忆经历了什么处理。

### 记忆生成与语义无损压缩

生成接口负责根据历史数据和关注的切面来产出记忆。这里需要区分两个概念：memory 是生成阶段的产物，而 memoryData 是持久化到数据库之后的结构。生成阶段使用 memory，存储时才转为 memoryData。

生成的核心工作是语义无损压缩，分为三个步骤：首先去除指示代词（如"这个""那个"），然后去除模糊的时间表述，最后明确实体之间的关系。目标是确保每一条记忆都可以脱离原始上下文独立使用。这套做法参考了之前调研的语义压缩库的方案。

### 时间处理机制

生成接口有一个"生成时间"参数，用于在提示词中附加当前时间信息（例如"2026 年 2 月 9 号"），让模型在生成记忆时有时间参照。具体的时间推断逻辑分三种情况：

- 如果记忆内容中没有提及任何时间，就以当前生成时间作为这条记忆的时间
- 如果记忆内容包含相对时间表述（如"两个月前让你做的事情"），则从当前时间往前推算，得出确切时间
- 如果记忆内容中明确给出了绝对时间（如"24 年 1 月 4 号做的事情"），就直接采用这个明确的时间

最终确定的时间会写入 JSON Schema 输出结构中的一个专用字段，而不是混入记忆文本本身。这一点在测试中已经验证过——由于 JSON Schema 中对时间字段的格式有明确约束，模型不会将时间信息错误地嵌入到记忆正文中。不过，提示词中用于举例的示例结构需要做一个小调整：要么去掉不完整的示例，要么改为带完整结构的小型示例，避免对模型的输出产生误导。

## 模板库：从零重构

### 重构背景

模板库进行了彻底的重写，原有的所有代码全部删除。之前的架构只有两层：将模板解析为语法树，然后根据语法树生成最终的字符串。重构后增加到三层处理流水线。

### 三层架构

第一层是词法分析器。它接收一段模板文本，将其拆分为最小的词法单元——文本片段和符号标记各自独立。第二层是语法分析，基于词法分析器的输出构建语法树。语法树中的每个节点现在都携带了位置信息。第三层是执行阶段，遍历语法树生成最终结果。

### 精确的错误报告

新增的位置信息是这次重构的一个重要改进。当模板存在语法错误时（比如少了一半花括号），报错信息会精确告诉你：在第几行、第几个字符、是什么标签、出了什么问题。这对于 AI 辅助修复特别有价值——只需让 AI 运行一下这个模板，解析阶段就会直接抛出精确的错误信息，AI 能够据此定位并修正问题。

### 注册器模式的扩展机制

架构上引入了注册器模式，与之前过滤器的设计保持一致。如果需要扩展新的标签（比如添加一个 set 标签），步骤非常清晰：

1. 定义 set 节点的结构
2. 实现两个方法——节点的执行方法和节点标签的解析方法
3. 将实现注册到注册函数中

完成这三步，模板库就具备了新标签的能力，完全不需要修改已有代码。

不过当前的注册是全局级别的，这在一个业务系统中多处使用模板库时会产生问题——不同业务方注册的扩展标签会互相影响。if 这类基础语法标签做全局注册没问题，但业务方自定义的扩展标签应该只在自己的作用域内生效。这一点需要调整为实例级别的注册，避免全局污染。

## PDF 处理方案：技术评估与路径选择

### 两个库的对比分析

对一个第三方 PDF 库和 Google 的 PDF 库进行了详细对比。第三方库在某些方面做得比 Google 的库更好——它具备 PDF 修复能力，能够将损坏的 PDF 修复后再解析。考虑到 Google 的库已经将近十年没有更新，第三方库在功能丰富度上有明显优势。

但两个库的侧重点不同。第三方库更偏向底层完整解析和文档修复，而 Google 的库更像是面向 AI 应用场景的 PDF 工具——它关注的是页面内容的编辑操作，但不提供元数据操作、PDF 修复、安全加密等底层功能。如果将两个库互补，几乎可以覆盖所有 PDF 相关的功能。

### 决策：渐进式推进

当前的策略是先基于 Google 的库实现编辑功能（页面内容修改、添加页面、删除页面等面上的功能），暂时不追求底层的 PDF 修复和安全加密。等到 AI 成本进一步下降（再降十倍左右），再考虑参考第三方库的架构从零改写一个完整的 PDF 处理库。第三方库的处理流程非常清晰，作为参考蓝图没有问题，只是当前逐步改写的人力和 AI 成本都还太高。

# 文档转换能力与检索结构构建

本次讨论围绕文档转换能力的近期实现进展展开，核心目标是把已有的分步能力沉淀为可用的转换流程，同时保持模块解耦、便于测试和后续优化。讨论覆盖了测试策略、doc/docx 支持、markdown 转换策略、PDF 目录与页码映射方案、整合方式与命名设计方法。

## 总体结论

当前阶段已经完成了多个关键模块的单点实现，包含 doc/docx 创建能力、部分解析能力、markdown 转换能力，以及 PDF 目录处理的四步流程。下一阶段重点不在"再加新能力"，而在"把已实现能力稳定组装起来"，并通过 pipeline 方式接入业务，处理多次模型调用带来的状态管理和失败恢复问题。

## 讨论主线与设计思路

### 1. 转换接口与摘要能力解耦

当前转换接口读取结果由两部分组成：

- node 切片：用于承载文档主结构
- metadata：用于承载元信息

node 采用切片是因为文档结构里可能存在多个同级标题节点，需要天然支持多根或并列结构。

在接口中预留了 summary 字段，但当前转换层不会填充每个节点摘要。摘要生成被明确拆分到转换后阶段处理。

这样设计的原因是：

- 转换层职责聚焦在"结构提取与内容提取"
- 摘要层职责聚焦在"语义压缩与概括"
- 两者解耦后，摘要模块可以独立迭代，不会反向影响基础解析稳定性

此外，内容在转换阶段已经完整提取，后续只需要对每个节点调用现有摘要模块即可。

### 2. Markdown 转换实现与层级压缩策略

markdown 转换已完成，用法保持简洁：调用转换即可得到统一结构。

真正复杂点不在调用，而在标题判断：

- 标题判断基于"按行识别"
- markdown 实际内容中混有 html 与非标准写法
- 标准中存在会干扰判断的规则场景
- 已通过正则把这些干扰项定义并过滤，减少误判

另一个关键策略是"减枝/压缩"：

- 风险是文档层级过深导致结构过碎
- 解析天然可到 h1 到 h6 共 6 层
- 不采取"固定压到 3 层"的硬策略
- 采用"子节点 token 阈值"策略：当子节点过小，向上合并；当节点足够大，不触发压缩

该策略是为同时兼容两类文档：

- 大文档（如接近一本书的体量）：若强行压缩会破坏已合理的大节点结构
- 小文档：容易出现层级多但内容薄，适合压缩避免拆得过散

### 3. PDF 处理难点与四步方案

PDF 侧复杂度最高，核心问题是"目录识别"与"逻辑页码到物理页码映射"。目前已拆成四个模块并完成实现。

第一步：目录存在性检测。在前若干页内检测是否存在目录。使用阈值控制检测范围。若前置页未检出目录，默认该 PDF 无目录，转入目录生成流程。

第二步：目录结构提取。当目录存在时，从内容中提取目录结构。因原始格式不规整，需要模型辅助提取。提取目标包含目录等级、标题、页码。

第三步：目录生成。当第二步前提不成立（即没有可提取目录）时，使用生成器生成目录。

第四步：页码映射转换。将逻辑页码转换为物理页码。原因是 PDF 目录页码通常从正文某处起算，前置页未计入。若不完成映射，后续内容抽取无法与原始 PDF 页面对齐。

四步之后仍需做目录页码校验。参考库采用"页码不对就从头重生成"的策略，但该方式成本很高。当前实现已完成核心步骤，后续会在校验与低成本纠错上继续优化。

### 4. 整合策略、业务接入与状态管理

现状是各步骤已分散实现，但"一体化组装"尚未完成。当前计划是在统一位置组装流程，用于验证端到端效果。

同时讨论确认：

- 这套组装更适合先作为能力示例，而不是直接作为业务最终编排
- 业务侧更可能通过 pipeline 组装，因为每一步都涉及大模型调用
- 多步链路中即使下一步失败，上一步状态仍有价值，不能简单丢弃

这意味着后续编排需要重点支持：

- 中间状态保留
- 失败后的可恢复执行
- 分步重试而非全链路重跑

## 下一步实施重点

### 示例分析

这个示例展示了以下特点：

1. **明确的总体结论**：开篇就说明了当前状态和下一步重点
2. **按主题组织**：每个主题都有清晰的小标题和逻辑结构
3. **解释设计原因**：不仅说明了"做什么"，还解释了"为什么这样设计"
4. **技术细节适度**：提供了足够的技术细节，但不过度展开
5. **清晰的进展清单**：用"已完成"、"进行中"、"待推进"清晰地呈现当前状态
6. **可执行的后续计划**：明确了下一步的优先级和具体任务

## 两个示例的共同特点

1. **完整的 frontmatter**：包含所有必需字段，topics 使用具体的关键词
2. **清晰的开篇**：用 1-2 段概述会议的核心内容和目标
3. **结构化的主体**：使用清晰的标题层级组织内容
4. **解释决策原因**：不仅说明做什么，还解释为什么
5. **统一的叙述视角**：不提及具体发言人，以统一的视角呈现
6. **连贯的语言**：避免过度使用列表，保持段落的连贯性
7. **明确的后续计划**：说明当前状态和下一步行动

### 这个示例的问题

1. **frontmatter 不规范**：topics 太泛，description 是模板句
2. **包含无用信息**：会议时间、地点、参与人等信息对后续决策没有价值
3. **对话式记录**：逐句转录对话，没有提炼核心观点
4. **缺少技术细节**：没有说明为什么要重构，三层架构是什么
5. **缺少决策背景**：没有说明为什么选择 Google 的库
6. **结论空洞**：没有具体的后续行动项
