# 会议文档写作原则

本文档详细说明会议文档撰写的核心原则，确保生成的文档清晰、有用、专业且避免过度文档化。

## Clear（清晰）

### 技术准确性与可读性并重

会议文档需要准确传达技术细节，同时保持可读性。避免两个极端：

- **过于简化**：丢失关键技术细节，导致后续无法追溯决策依据
- **过于复杂**：堆砌术语和实现细节，让非专业读者无法理解核心思路

**良好示例**：

```markdown
记忆库移除了之前的"演化"功能，当前聚焦于三个核心能力：生成、合并和过期淘汰。
其中合并和生成沿用了之前的接口设计，只是将它们统一迁移到了 memory 模块内部。
```

这段文字既说明了架构调整（移除演化功能），又解释了当前的核心能力，同时指出了与之前版本的关系。

### 使用明确的术语

在整篇文档中保持术语一致性：

- 如果使用"记忆库"，就不要混用"memory 模块"、"记忆系统"等不同表述
- 如果使用"合并"，就不要在其他地方说"归并"、"融合"
- 技术术语首次出现时可以用括号注明英文原文，如"记忆库（Memory）"

### 确保非技术背景读者能理解核心内容

虽然会议文档面向技术团队，但应该让产品经理、项目经理等非技术角色也能理解核心决策：

- 解释技术选择的业务价值
- 说明技术限制对产品的影响
- 用类比或示例帮助理解抽象概念

**良好示例**：

```markdown
第三方库采用的是 AGPL 许可证，这意味着依赖它会要求整个项目也必须开源，
因此无法直接使用。
```

这段话不仅说明了许可证类型，还解释了它对项目的实际影响。

## Useful（有用）

### 只添加对理解有价值的内容

避免记录无关紧要的细节：

**不必要的内容**：
- "会议于下午3点开始，持续了2小时"
- "张三提出了一个想法，李四表示赞同"
- "经过激烈讨论后，大家达成了共识"

**有价值的内容**：
- 讨论的技术背景和问题
- 不同方案的优劣对比
- 最终决策的原因
- 待解决的问题和风险

### 聚焦于"为什么"而非仅仅"是什么"

**不够好的写法**：

```markdown
模板库进行了重构，新增了词法分析器。
```

**更好的写法**：

```markdown
模板库进行了彻底的重写，原有的所有代码全部删除。之前的架构只有两层：
将模板解析为语法树，然后根据语法树生成最终的字符串。重构后增加到三层处理流水线。

第一层是词法分析器。它接收一段模板文本，将其拆分为最小的词法单元——
文本片段和符号标记各自独立。这样做的原因是为了在后续阶段提供精确的位置信息，
当模板存在语法错误时，能够准确告诉用户在第几行、第几个字符出了什么问题。
```

第二个版本不仅说明了"做了什么"，还解释了"为什么这样做"。

### 提供可追溯的决策背景和影响分析

每个重要决策都应该包含：

1. **背景**：为什么需要做这个决策
2. **选项**：考虑了哪些方案
3. **权衡**：每个方案的优劣
4. **决策**：最终选择了什么
5. **影响**：这个决策会带来什么影响

**完整示例**：

```markdown
### 决策：渐进式推进

当前的策略是先基于 Google 的库实现编辑功能（页面内容修改、添加页面、
删除页面等面上的功能），暂时不追求底层的 PDF 修复和安全加密。

这个决策基于以下考虑：

1. Google 的库虽然功能有限，但可以满足当前的核心需求
2. 第三方库的 AGPL 许可证限制了商业使用
3. 从零实现完整的 PDF 处理库需要 67 个步骤，当前的人力和 AI 成本都太高

等到 AI 成本进一步下降（再降十倍左右），再考虑参考第三方库的架构从零改写
一个完整的 PDF 处理库。第三方库的处理流程非常清晰，作为参考蓝图没有问题。
```

## Professional（专业）

### 技术但不压倒性

使用准确的技术术语，但不要堆砌术语来显示专业性：

**过度技术化**：

```markdown
通过实现基于 AST 的词法分析器和语法分析器，构建了一个三阶段的编译流水线，
利用访问者模式遍历语法树节点，实现了模板的解释执行。
```

**更平衡的表述**：

```markdown
新架构分为三层：词法分析、语法分析和执行。词法分析器将模板文本拆分为
最小单元，语法分析器构建语法树，执行阶段遍历语法树生成最终结果。
```

### 自信但不自夸

陈述事实和决策，不要过度评价：

**过度自夸**：
- "这是一个非常优雅的解决方案"
- "我们采用了业界最先进的架构"
- "这个设计完美地解决了所有问题"

**客观陈述**：
- "这个方案能够解决 X 问题，同时保持了 Y 的灵活性"
- "参考了 Z 的架构设计，并根据我们的需求进行了调整"
- "当前设计解决了主要问题，但仍有 X 需要后续优化"

### 保持客观中立的叙述视角

不要提及具体的发言人，使用统一的叙述视角：

**不好的写法**：

```markdown
张三提出应该使用方案A，但李四认为方案B更好。经过讨论，王五建议采用方案C。
```

**更好的写法**：

```markdown
团队评估了三种方案。方案A的优势在于实现简单，但扩展性有限。方案B扩展性好，
但需要较大的重构成本。最终选择了方案C，它在实现复杂度和扩展性之间取得了平衡。
```

## Avoid Over-documentation（避免过度文档化）

### 融入关键见解，而非搬运会议记录

会议文档不是会议记录的逐字稿，而是提炼后的知识沉淀：

**会议记录风格**（避免）：

```markdown
A: 我觉得我们应该用三层架构
B: 为什么要三层？
A: 因为可以提供更好的错误信息
B: 那第一层是什么？
A: 词法分析
B: 第二层呢？
A: 语法分析
```

**会议文档风格**（推荐）：

```markdown
新架构采用三层设计，主要是为了提供精确的错误报告。第一层词法分析将模板
拆分为最小单元，第二层语法分析构建语法树并记录位置信息，第三层执行阶段
生成最终结果。当模板存在错误时，系统能够精确指出错误位置。
```

### 不要逐句转录对话

对话式的内容需要转化为连贯的叙述：

**对话式**（避免）：

```markdown
Q: 为什么要移除演化功能？
A: 因为当前不需要这个功能了。

Q: 那现在有哪些核心功能？
A: 生成、合并和过期淘汰。
```

**叙述式**（推荐）：

```markdown
记忆库移除了之前的"演化"功能，当前聚焦于三个核心能力：生成、合并和过期淘汰。
```

### 提炼核心论点和决策逻辑

每个段落应该有明确的主题，围绕一个核心观点展开：

**结构松散**（避免）：

```markdown
我们讨论了模板库的问题。现在的实现有一些不足。错误信息不够清楚。
我们决定重构。新的架构会更好。会有三层。第一层是词法分析。
```

**结构清晰**（推荐）：

```markdown
### 精确的错误报告

新增的位置信息是这次重构的一个重要改进。当模板存在语法错误时（比如少了
一半花括号），报错信息会精确告诉你：在第几行、第几个字符、是什么标签、
出了什么问题。这对于 AI 辅助修复特别有价值——只需让 AI 运行一下这个模板，
解析阶段就会直接抛出精确的错误信息，AI 能够据此定位并修正问题。
```

## 语言风格指南

### 段落组织

- 每个段落聚焦一个核心观点
- 段落开头用主题句概括要点
- 后续句子提供支撑细节或解释
- 段落之间保持逻辑连贯

### 句子结构

- 优先使用主动语态："团队选择了方案A"而非"方案A被选择"
- 避免过长的句子，一个句子表达一个完整的意思
- 适当使用短句来强调重点

### 列表使用

**适合使用列表的场景**：
- 并列的多个要点
- 步骤或流程
- 配置项或参数说明
- 待办事项或进展清单

**不适合使用列表的场景**：
- 需要解释因果关系的内容
- 需要展开论述的观点
- 有逻辑递进关系的内容

### Markdown 格式

- 不要过度使用粗体强调，只在关键术语或重要概念处使用
- 代码块用于代码示例、配置示例、命令行示例
- 引用块用于重要的结论或警告信息
- 表格用于对比不同方案或列举参数

## 常见问题

### 如何处理技术细节的深度？

根据内容的重要性和复杂度决定：

- **核心决策**：详细解释背景、原因、影响
- **实现细节**：提供足够的上下文，但不需要代码级别的细节
- **配置参数**：说明关键参数的作用，不需要列举所有参数

### 如何平衡完整性和简洁性？

- 保留所有影响决策的关键信息
- 省略过程中的试错和反复讨论
- 将最终的结论和原因清晰呈现
- 对于复杂的技术细节，可以说"详见 X 文档"而不是全部展开

### 如何处理未达成共识的内容？

- 明确说明存在分歧
- 列出不同的观点和理由
- 说明当前的临时决策或待定状态
- 记录需要后续讨论的问题

**示例**：

```markdown
### 注册器模式的作用域

当前的注册是全局级别的，这在一个业务系统中多处使用模板库时会产生问题——
不同业务方注册的扩展标签会互相影响。if 这类基础语法标签做全局注册没问题，
但业务方自定义的扩展标签应该只在自己的作用域内生效。这一点需要调整为
实例级别的注册，避免全局污染。
```

## 检查清单

在完成文档后，使用以下清单进行自查：

### 清晰性
- [ ] 技术术语使用准确且一致
- [ ] 非技术读者能理解核心决策
- [ ] 每个段落有明确的主题
- [ ] 逻辑连贯，前后呼应

### 有用性
- [ ] 保留了所有关键决策信息
- [ ] 解释了"为什么"而非仅仅"是什么"
- [ ] 提供了可追溯的决策背景
- [ ] 明确了影响和后续行动

### 专业性
- [ ] 使用准确的技术术语但不过度堆砌
- [ ] 客观陈述事实，不过度评价
- [ ] 统一的叙述视角，不提及具体发言人
- [ ] 语言专业但易于理解

### 避免过度文档化
- [ ] 没有逐字转录对话
- [ ] 提炼了核心观点和决策逻辑
- [ ] 段落结构清晰，不是流水账
- [ ] 适度使用列表和格式，不过度装饰
