# 会议文档撰写

将会议讨论内容转化为详细、完整、结构清晰的会议文档，保留所有关键细节，深入阐述设计决策的原因，并以统一的叙述视角呈现。

## 核心目标

生成的会议文档应当：

1. **保留所有细节** - 不遗漏任何讨论内容，尽可能保持原来的会话信息
2. **解释是什么和为什么** - 深入阐述设计决策的原因和背景
3. **整理讨论思路** - 将对话式内容转化为清晰的逻辑思路
4. **遵循写作原则** - 使用清晰、专业、有用的语言
5. **统一叙述视角** - 不提及发言人，以统一的客观陈述视角呈现
6. **采用系统阐述方式** - 重点回答"架构是什么、为什么这样设计、会带来什么影响"，避免停留在"会议里发生了什么"

## 篇幅要求

文档篇幅需与会议时长和讨论深度匹配，确保信息不被过度压缩。

**计算公式：** `最少行数 = ceil(会议分钟 / 10) × 78`

常见时长对照：

| 会议时长 | 最少正文行数（不含 frontmatter） |
|---------|-------------------------------|
| 10 分钟 | 78 行 |
| 20 分钟 | 156 行 |
| 30 分钟 | 234 行 |
| 40 分钟 | 312 行 |
| 50 分钟 | 390 行 |
| 60 分钟 | 468 行 |
| 90 分钟 | 702 行 |
| 120 分钟 | 936 行 |
| 180 分钟 | 1404 行 |
| 240 分钟 | 1872 行 |

必须通过扩展讨论细节、方案对比、决策背景来达到篇幅，不得通过空行、列表拆分等排版手段凑行数。

篇幅不足通常意味着讨论细节被遗漏。当篇幅不够时，回顾原始内容，检查是否有以下遗漏：

- 方案对比的具体优劣势描述不够充分
- 技术决策的背景和原因缺少展开
- 讨论中提到的具体例子、场景、约束条件被省略
- 后续计划和行动项缺少足够的上下文

**文字连贯性要求：**

- 优先使用连贯的段落叙述，避免过度碎片化的短句和频繁换行
- 每个段落最少包含 3 个完整句子，形成完整的论述单元
- 减少单句成段的情况，将相关观点合并为连贯段落
- 列表仅用于并列信息、步骤说明或清单，不应成为主要表达方式
- 段落之间使用自然的过渡，保持叙述流畅性

## 文档结构要求

### Frontmatter 规范

每篇会议文档必须包含完整的 YAML frontmatter：

```yaml
---
title: "与文件名主题完全一致"
date: "YYYY-MM-DD"
categories:
  - "前端组 | 后端组 | 产品设计组 | 全员（四选一）"
topics:
  - "具体关键词1"
  - "具体关键词2"
description: "2-3句具体描述，写清讨论对象、关键决策、影响范围。"
---
```

**字段要求：**

- `title`：与文件名主题完全一致（不含日期前缀）
- `categories`：只能有一个值（单一组别）
- `topics`：必须是关键词，不是动作句。避免泛词如"AI"、"会议纪要"、"产品设计"、"交互设计"、"架构设计"。推荐使用模块名、机制名、协议名、组件名、策略名等名词短语
- `description`：必须 2-3 句，明确讨论对象、核心决策或方案取舍、预期影响。禁止使用"围绕《xxx》展开，记录核心讨论点..."这类模板句

### 文件命名规范

格式：`YYYYMMDD-主题.md`

主题用于生成标题，必须与文档标题一致。

### 标题与正文结构

- 文档首个标题 `# H1` 必须与文件名主题完全一致
- 禁止重复标题（例如 frontmatter 后又出现第二个等价主标题）
- 清理模板化"会议时间/时长/参与人员"等冗余头部块，正文直接进入内容
- 不要太多 markdown 强调语法（如过度使用粗体），保持语句连贯

## 写作原则

参考 `references/writing-principles.md` 中的详细指导原则。核心要点：

### Clear（清晰）

- 保持技术准确性的同时确保可读性
- 使用明确的术语，避免模糊表述
- 确保非技术背景读者也能理解核心内容

### Useful（有用）

- 只添加对用户理解有价值的内容
- 聚焦于"为什么"而非仅仅"是什么"
- 提供可追溯的决策背景和影响分析

### Professional（专业）

- 技术但不压倒性
- 自信但不自夸
- 保持客观中立的叙述视角

### Avoid Over-documentation（避免过度文档化）

- 融入关键见解，而非搬运会议记录
- 不要逐句转录对话
- 提炼核心论点和决策逻辑

### 语言风格约束（必须）

- 去掉明显口述纪要腔，例如"会议指出""讨论中提到""团队认为"等起手句。
- 默认使用客观陈述式写法，直接陈述事实、判断、机制与影响，不用"谁说了什么"的转述框架。
- 叙述重点从"记录发生了什么"转为"系统性阐述架构是什么、为什么这样设计、会带来什么影响"。
- 保留结构主线，但将偏短段落扩展为完整论述单元（背景、机制、取舍、影响至少覆盖其中三项）。
- 避免大量单句换行，优先形成信息密度更高、过渡自然的连贯段落。

## 文档撰写流程

### 1. 理解会议内容

仔细阅读原始会议记录或讨论内容，识别：

- 讨论的核心主题和子主题
- 关键技术决策及其原因
- 不同方案的权衡与取舍
- 达成的共识和待解决的问题
- 后续行动项

### 2. 规划文档结构

根据讨论内容选择合适的组织方式：

**按主题分组**（适用于多个独立议题）：
```markdown
# 会议主题

## 主题一：XXX系统架构设计

### 整体架构调整
### 核心模块设计
### 技术选型

## 主题二：YYY功能实现方案

### 方案对比
### 最终决策
```

**按时间线组织**（适用于单一主题的深入讨论）：
```markdown
# 会议主题

## 背景与问题
## 方案探讨
## 技术细节
## 决策与结论
## 后续计划
```

**按决策层次组织**（适用于复杂决策过程）：
```markdown
# 会议主题

## 总体结论
## 讨论主线与设计思路
### 1. 核心问题
### 2. 方案对比
### 3. 最终决策
## 当前进展清单
## 下一步实施重点
```

### 3. 撰写内容

**开篇段落**：
- 用 1-2 段概述会议的核心内容
- 说明讨论的背景和目标
- 不要使用"本次会议讨论了..."这类模板句

**主体内容**：
- 使用清晰的小标题组织内容
- 每个段落聚焦一个核心观点，最少包含 3 个完整句子
- 解释技术决策的"为什么"，不仅仅是"是什么"
- 优先使用连贯段落叙述，将相关观点串联成流畅的论述
- 避免单句成段和过度碎片化的短句
- 列表仅用于并列信息、步骤说明或清单，不应替代段落叙述

**技术细节**：
- 对于复杂的技术方案，提供足够的上下文
- 说明不同方案的优劣势
- 解释最终选择的原因

**结论部分**：
- 总结已完成的工作
- 列出进行中的任务
- 明确下一步行动项

### 4. 优化与检查

**语言优化**：
- 去除口语化表述
- 统一术语使用
- 确保逻辑连贯
- 减少不必要的强调语法（粗体、斜体等）

**中英文混排规范**：
- 中文与英文之间必须增加空格
  - 正确：`使用 Agent 功能可以自动处理任务`
  - 错误：`使用Agent功能可以自动处理任务`
- 中文与数字之间必须增加空格
  - 正确：`我们提供了 5 种不同类型的 Agent`
  - 错误：`我们提供了5种不同类型的 Agent`
- 数字与单位之间不需要增加空格
  - 正确：`文件大小不超过 2MB`
  - 错误：`文件大小不超过 2 MB`
- 全角标点与其他字符之间不加空格
  - 正确：`Agent 已成功创建！`
  - 错误：`Agent 已成功创建 ！`
- 中文逗号后面必须增加空格(提升可读性)
  - 正确：`团队整理了规范, 并将最佳实践封装为技能包`
  - 错误：`团队整理了规范,并将最佳实践封装为技能包`
- 专有名词使用正确的大小写
  - 正确：`使用 GitHub 账号登录`
  - 错误：`使用 github 账号登录`
- 代码、命令、文件路径使用反引号标记
  - 正确：`使用 \`git submodule\` 命令`
  - 错误：`使用 git submodule 命令`

**结构检查**：
- 验证 frontmatter 格式正确
- 确认标题层级合理
- 检查是否有重复或冗余内容

**质量检查**：
- 是否保留了所有关键信息
- 是否解释了决策背景
- 是否易于理解和检索
- 是否符合团队文档规范
- 是否符合中英文混排规范

## 常见模式

### 技术方案讨论

```markdown
## XXX技术方案选型

### 方案对比

当前面临的核心问题是[问题描述]。团队评估了三种可能的方案。方案A采用[技术栈]，优势在于[优势]，但存在[劣势]的问题。方案B基于[技术栈]，能够解决[问题]，但需要[代价]。方案C则[描述]，在某些场景下具有独特优势。

### 最终决策

经过权衡，选择了方案B。主要考虑因素包括三个方面：首先是[原因1]，这直接影响了系统的长期可维护性。其次是[原因2]，能够显著降低团队的学习成本。最后是[原因3]，与现有技术栈的兼容性更好。虽然方案B需要[代价]，但从长期来看[长期价值]，这个投入是值得的。
```

### 架构设计讨论

```markdown
## XXX架构重构

### 重构背景

现有架构存在[问题描述]。随着[业务发展/技术演进]，这些问题逐渐凸显，影响了系统的可扩展性和维护效率。

### 新架构设计

新架构采用[设计模式/架构风格]，分为三个核心层次。第一层是[层次名称]，它负责[职责]，通过[机制]实现[目标]。第二层是[层次名称]，它基于第一层的输出进行[处理]，确保数据流转的高效性。第三层是[层次名称]，它[职责描述]，为上层应用提供统一接口。

### 关键改进

这次重构带来了几个重要改进。首先是[改进1]，这使得[价值]，显著提升了系统响应速度。其次是[改进2]，解决了之前[问题]，降低了运维复杂度。此外，新架构在扩展性方面也有明显提升，为后续功能迭代预留了充足空间。

### 待解决问题

当前设计仍有[问题]需要后续优化。具体来说，[详细描述]，这部分将在下一个迭代周期中重点攻克。
```

### 进展同步

```markdown
## 当前进展清单

已完成：

- [功能/模块]的核心实现
- [功能/模块]的测试覆盖
- [文档/规范]的初版

进行中：

- [任务]的开发，预计[时间节点]完成
- [问题]的调研与方案设计

待推进：

- [任务]需要在[前置条件]完成后启动
- [优化项]可在后续迭代中处理
```

## 质量标准

一份高质量的会议文档应当：

- ✅ 完整保留所有关键讨论内容
- ✅ 清晰解释技术决策的背景和原因
- ✅ 结构清晰，易于检索和理解
- ✅ 语言专业但不晦涩
- ✅ 符合团队文档规范
- ✅ 为未来的决策提供可追溯的依据
